## 需求清单（步骤 1）

### 功能需求（PRD §7）

| ID | 摘要 |
|----|------|
| FR-A1 | 中国大陆手机号 + 6 位验证码后进入后续流程 |
| FR-A2 | 客户端持久化档案（手机号、昵称、出生年月日、星座） |
| FR-A3 | 注销并清除持久化档案 |
| FR-H1 | 由出生月、日计算西方星座 |
| FR-H2 | 用出生日期、星座中文名、日历日期向服务端请求当日运势 |
| FR-H3 | AI 不可用时展示兜底运势 |
| FR-H4 | 运势字段：启示、幸运色（名+hex）、幸运数字、幸运物品、恋爱/事业/能量（星级+短句） |
| FR-T1 | 树洞为群聊（共享房间），非一对一私聊 |
| FR-T2 | 可滚动消息列表与会话内发送文本 |
| FR-T3 | 分类筛选可过滤展示，不得破坏发送/浏览 |
| FR-M1 | 明确操作「开始匹配」进入队列 |
| FR-M2 | 至少两人时配对并移出队列 |
| FR-M3 | 配对随机 |
| FR-M4 | 实名认证非匹配必要条件 |
| FR-M5 | 配对后展示匹配结果（如对方卡片） |
| FR-N1 | 已登录用户底部导航：运势 / 树洞 / 匹配 / 我的 |
| FR-N2 | 壳标题与品牌一致（星聊） |

### 非功能需求（PRD §8）

| ID | 摘要 |
|----|------|
| NFR1 | 档案摘要中手机号掩码（中间位） |
| NFR2 | AI 密钥不得打入前端包 |
| NFR3 | 运势首屏 90% 用户 45 秒内有效内容；客户端单次请求超时 30 秒；超时或 5xx 走兜底 |
| NFR4 | 运势 JSON 结构文档化、前后端一致 |
| NFR5 | 主要文案中文 |
| NFR6 | 四浏览器近两版；主路径可键盘；WCAG 2.1 AA 为增强目标 |

### 架构文档补充（与实现相关）

- 新建 **ASP.NET Core** 工程（建议 `server/AstralVeil.Api/`），实现 **`POST /api/horoscope`**，与 PRD §14 契约一致；**OpenAPI 3**、错误码稳定、**camelCase** JSON。
- 开发环境 **Vite 代理** `/api` → .NET；`OPENAI_API_KEY` 仅服务端 / User Secrets。
- 运势端点 **限流**（公开接口）；MVP 档案仍为 **localStorage**；增长期再定 PostgreSQL / JWT 等。

### UX 设计文档

**无** 独立 UX 稿；验收以 PRD 旅程 §6 与上述 FR/NFR 为准。

## Epic 与 Story 详情（步骤 3）

### Epic 1：可靠每日运势

**目标：** 满足 FR-H1～FR-H4 及 NFR2、NFR3、NFR4；生产运势由 **.NET** 托管（与架构一致），开发期可经代理联调。

#### Story 1.1 — 星座推算与请求载荷

**用户故事：** 作为已建档用户，我希望系统根据我的出生月日算出星座，并在请求运势时带上 PRD 约定的字段，以便获得「我的」当日运势。

**验收标准（Given / When / Then）：**

- **Given** 用户在建档流程中已选择或输入合法出生年月日  
- **When** 进入运势页或触发运势刷新  
- **Then** 客户端计算的星座中文名与 PRD 西方星座规则一致，且 `POST /api/horoscope` 请求体包含 `birthDate`、`zodiac`、`date`（及架构约定的 camelCase 字段）

**覆盖：** FR-H1、FR-H2、NFR4（与 §14 示例一致）

---

#### Story 1.2 — ASP.NET Core 运势 API

**用户故事：** 作为运维/产品，我希望生产环境由 **ASP.NET Core** 提供 `POST /api/horoscope`，以便密钥留在服务端且契约可文档化。

**验收标准：**

- **Given** 仓库内存在 `server/AstralVeil.Api/`（或架构约定等价路径）  
- **When** 向 `POST /api/horoscope` 发送 PRD §14.1 合法 JSON  
- **Then** 成功时返回 §14.2 结构（camelCase）；失败时返回 §14.3 风格 `error` 码；**OpenAPI** 可访问；密钥不进入前端构建

**覆盖：** FR-H2、FR-H4（服务端生成或校验结构）、NFR2、NFR4；架构 §4.3

---

#### Story 1.3 — 开发代理与前端超时兜底

**用户故事：** 作为开发者/用户，我希望本地前端请求 `/api` 转发到 .NET，且浏览器端在 30 秒超时或 5xx 时看到兜底运势，避免无限加载。

**验收标准：**

- **Given** `vite.config`（或等价）配置 `/api` 代理至本地 API 基址  
- **When** 运势请求超过 30 秒或收到 5xx/网络错误  
- **Then** UI 展示 FR-H3 兜底内容，不出现无限转圈；与 PRD 旅程 §6.1 步骤 4 一致

**覆盖：** FR-H3、NFR3；架构 4.3 / 4.4

---

#### Story 1.4 — 限流与健康检查（最小）

**用户故事：** 作为服务方，我希望公开运势接口具备基本限流与健康检查，降低滥用风险。

**验收标准：**

- **Given** API 部署配置可用  
- **When** 高频重复调用或探活  
- **Then** 存在可文档化的限流策略（即使为开发级 IP/固定窗口）；`/health` 或等价返回 200

**覆盖：** 架构 §2.2、§4.3（MVP API 限流）

---

### Epic 2：账号与档案

**目标：** FR-A1～FR-A3、NFR1。

#### Story 2.1 — 手机号与验证码门禁

**用户故事：** 作为新用户，我只有在输入合法手机号与 6 位验证码后才能进入建档，以减少无效流程。

**验收标准：**

- **Given** 用户在登录页  
- **When** 手机号非 11 位或验证码非 6 位  
- **Then** 无法进入下一步或展示明确校验提示（与现有 UI 行为一致或可测）

**覆盖：** FR-A1

---

#### Story 2.2 — 档案持久化与展示

**用户故事：** 作为回访用户，我希望昵称、生日、星座被记住；在「我的」中手机号以掩码显示。

**验收标准：**

- **Given** 用户完成建档  
- **When** 关闭并重新打开应用（同域同存储）  
- **Then** 档案仍存在；摘要中手机号为掩码格式（NFR1）

**覆盖：** FR-A2、NFR1

---

#### Story 2.3 — 注销

**用户故事：** 作为用户，我希望注销后本设备档案被清除。

**验收标准：**

- **Given** 用户已登录  
- **When** 在「我的」执行注销并确认  
- **Then** 持久化档案清除，回到登录/建档前状态（与 PRD 旅程 5 一致）

**覆盖：** FR-A3

---

### Epic 3：树洞群聊

**目标：** FR-T1～FR-T3。

#### Story 3.1 — 群聊房间与消息流

**用户故事：** 作为用户，我希望在树洞进入共享房间，滚动查看历史并发送消息，新消息在本会话内可见。

**验收标准：**

- **Given** 用户进入树洞 Tab  
- **When** 发送一条文本  
- **Then** 消息出现在列表中，且语义为群聊共享空间（非私聊）— FR-T1、FR-T2

**覆盖：** FR-T1、FR-T2

---

#### Story 3.2 — 分类筛选

**用户故事：** 作为用户，我希望按分类查看消息，且仍能正常发送与浏览列表。

**验收标准：**

- **Given** 存在多条带分类的消息  
- **When** 切换分类 Tab  
- **Then** 列表按策略过滤或限定展示，发送区仍可用— FR-T3

**覆盖：** FR-T3

---

### Epic 4：随机匹配

**目标：** FR-M1～FR-M5。

#### Story 4.1 — 入队

**用户故事：** 作为用户，我通过明确操作进入匹配队列。

**验收标准：**

- **Given** 用户在匹配页  
- **When** 点击「开始匹配」（或 PRD 等价主操作）  
- **Then** 进入等待/队列状态— FR-M1

**覆盖：** FR-M1

---

#### Story 4.2 — 配对与随机性

**用户故事：** 作为系统，当队列中至少两人时，应随机配对两人并移出队列；无需实名认证。

**验收标准：**

- **Given** 两个客户端（或双标签）均已入队  
- **When** 满足配对条件  
- **Then** 双方收到配对结果；配对对象在可重复测试中呈现随机性（演示语义与 PRD 一致）— FR-M2、FR-M3、FR-M4

**覆盖：** FR-M2、FR-M3、FR-M4

---

#### Story 4.3 — 匹配结果展示

**用户故事：** 作为用户，配对成功后我希望看到对方资料卡片（演示数据可接受）。

**验收标准：**

- **Given** 配对成功  
- **When** 展示结果屏  
- **Then** 可见匹配结果卡片（头像/昵称等占位）— FR-M5

**覆盖：** FR-M5

---

### Epic 5：应用壳与导航

**目标：** FR-N1、FR-N2；NFR6 基线。

#### Story 5.1 — 底部导航四 Tab

**用户故事：** 作为已登录用户，我希望通过底部导航在运势、树洞、匹配、我的之间切换。

**验收标准：**

- **Given** 用户已登录  
- **When** 点击各 Tab  
- **Then** 对应视图切换且无阻塞性错误— FR-N1

**覆盖：** FR-N1

---

#### Story 5.2 — 品牌标题

**用户故事：** 作为用户，我在壳层看到的产品名应为「星聊」。

**验收标准：**

- **Given** 应用在主壳层展示标题  
- **When** 查看顶栏/标题区  
- **Then** 与品牌「星聊」一致— FR-N2

**覆盖：** FR-N2

---

#### Story 5.3 — 主路径键盘可操作（基线）

**用户故事：** 作为仅使用键盘的用户，我希望主要流程可被操作（与 NFR6 一致）。

**验收标准：**

- **Given** 不依赖鼠标  
- **When** 完成登录 → 建档 → 进入运势主路径  
- **Then** 可通过 Tab/Enter 等完成；无不可达焦点陷阱（基线；完整 WCAG AA 为增强目标）

**覆盖：** NFR6
